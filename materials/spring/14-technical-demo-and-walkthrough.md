# 第14回 技術デモ・code walkthrough・知見整理

- 対象: B4・M
- 種別: 前期発展枠 / 固定

## この回の目標
- 自分の実装を、入力・処理・出力の流れに沿って 5 分程度で説明できる。
- 再実行に必要な情報と、他者に共有するには不足している情報を区別できる。
- 実装メモではなく、再利用可能な知見として何を残すべきかを判断できる。

## ミニ講義
- 技術デモでは「何を作ったか」よりも「どの入力に対して、どの処理を通し、何が出力されるか」を先に説明すると伝わりやすい。shape、保存物、実行コマンドはその骨格になる。
- code walkthrough では、全行を読む必要はない。処理の分岐点、前提条件、失敗しやすい箇所、再実行に必要な依存関係を押さえると、聞き手は自分で追試しやすくなる。
- 知見整理では、個人メモと共有資産を分けて考える。個人の一時的な試行錯誤と、他の学生が再利用できる説明や補助コードは価値が異なる。

## 演習
### 基礎レベル
1. 第02回から第13回までで自分が扱った script を 1 本選び、`session14_demo_notes.md` を作成する。
2. `session14_demo_notes.md` には次の 4 見出しをこの順で書く。
   - `## デモ対象`
   - `## 入力`
   - `## 出力`
   - `## 変更点`
3. `## デモ対象` には対象 script 名を 1 つ書く。
4. `## 入力` には入力ファイルまたは入力 tensor の shape、値域、前処理の有無を書く。
5. `## 出力` には保存物または logits / mask / embedding の shape と、どこを見れば成否を判断できるかを書く。
6. `## 変更点` には、処理の流れの中で自分が重要だと考える変更点を 3 行以内で書く。
7. `session14_demo_checklist.md` を作成し、次の 6 行をそのまま書き、各行の末尾に `OK` または `NG` を付ける。
```text
実行コマンドを説明できる
入力の shape を説明できる
出力の shape を説明できる
変更点を 3 行以内で説明できる
再実行に必要な file が揃っている
聞き手が次に見るべき file を示せる
```
8. `session14_demo_notes.md` の最後に 2 行追加し、なぜその script をデモ対象に選んだかを書く。単に「担当したから」ではなく、入力と出力の対応を説明しやすい理由を書く。

### 発展レベル
1. `session14_reuse_candidates.md` を作成し、次の 3 見出しをこの順で書く。
   - `## 共有してよいもの`
   - `## 共有前に手直しが必要なもの`
   - `## 共有しないもの`
2. 各見出しの下に 1 項目以上書き、項目ごとに「そう判断した理由」を 1 行添える。
3. B4 は `後輩に最初に伝えるべきこと` を 2 行書く。実行順序、つまずきやすい箇所、確認観点のうち 2 つ以上を含める。
4. M は `共有候補を採用するかどうかの判断基準` を 2 行書く。再利用性、保守負荷、誤解を招く危険のうち 2 つ以上を含める。
5. 最後に、`session14_demo_notes.md` と `session14_reuse_candidates.md` を見比べて、「デモとしては分かりやすいが共有資産としては弱い点」または「共有資産としては有用だがデモでは伝わりにくい点」を 2 行で書く。

## 確認ポイント
- `session14_demo_notes.md` に 4 見出しが順番通りあり、入力・出力・変更点が具体的に書かれている。
- `session14_demo_checklist.md` の 6 項目すべてに `OK` / `NG` が付いている。
- `なぜその script を選んだか` の説明が、入力と出力の対応に触れている。
- `session14_reuse_candidates.md` の各項目に判断理由があり、単なる分類だけで終わっていない。
- B4 または M の発展課題で、優先基準や判断基準が自分の言葉で書かれている。
