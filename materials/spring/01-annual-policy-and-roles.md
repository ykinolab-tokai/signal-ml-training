# 第01回 年間方針・役割分担・共通 repo・生成AI運用・コーディング規約

- 対象: B4・M
- 種別: 前期発展枠 / 固定

## この回の目標
- 年間 28 回の流れと、自分の学年で担う役割を説明できる。
- 共通 repo と `submission repo` の使い分けを、提出フローと結びつけて説明できる。
- 生成 AI を使ってよい場面と、申告が必要な理由を説明できる。

## ミニ講義
- この repo の `README.md` は年間方針の source of truth であり、学年ごとの責務、評価、提出先、生成 AI 方針は README を基準に読む。handout ごとのローカルな慣習よりも README を優先する。
- 共通 repo は教材、見本コード、共通資産を置く場所で、通常提出の置き場ではない。通常提出は各自の `submission repo` に残す。両者を混ぜると、教材更新と個人提出の履歴が衝突する。
- 役割分担では、B4 は extension を実装して後輩の視界を広げる側、M は品質確認と支援設計を担う側になる。単に「担当回を書く」だけでなく、自分がどこで価値を出すかまで決めると後続回が進めやすい。

## 演習
### 基礎レベル
1. `README.md` を読み、年間 28 回の構成、学年ごとの責務、通常提出と PR 練習の違い、生成 AI 利用時の申告ルールを確認する。
2. `session01_role_plan.md` を作成し、`## 自分の学年`, `## 前期で担当する回`, `## 後期で担う支援役割`, `## 今年度の重点` の 4 見出しを書く。各見出しには、README の記述に基づく自分の役割と判断理由を 1〜3 行で書く。
3. `session01_ai_ops_checklist.md` を作成し、`## 共通 repo の役割`, `## submission repo の役割`, `## 生成AI利用時の申告項目`, `## 今年気をつける運用ルール` の 4 見出しを書く。共通 repo と submission repo を混同すると何が困るかも 1 例入れる。
4. 最後に、今年度の自分の重点が `実装品質`, `説明品質`, `レビュー品質`, `再利用可能資産` のどれに近いかを選び、なぜその重点が後輩の学習に効くのかを 3 行以内で説明する。

### 発展レベル
1. B4 は `session01_role_plan.md` に `## extension 候補` を追加し、試したい発展課題 1 件と、その前に後輩が理解しているべき前提 2 項目を書く。
2. M は `session01_role_plan.md` に `## 介入条件` を追加し、mentor/reviewer として介入すべき条件を 3 項目書く。各項目には、介入が早すぎる場合と遅すぎる場合の問題を 1 行ずつ添える。
3. 生成 AI 利用申告が必要な場面を 2 例、不要な場面を 1 例書き、違いを `再現性` と `責任の所在` の観点から説明する。

## 確認ポイント
- `session01_role_plan.md` に 4 見出しが順番通りあり、担当回と支援役割が具体的に書かれている。
- `session01_ai_ops_checklist.md` に 4 見出しが順番通りあり、`用途` と `採用箇所` が明記されている。
- B4 は extension 候補と前提条件、M は介入条件と理由を書いている。
- repo の使い分けと AI 申告の違いが、単なる用語説明ではなく運用上の理由まで含めて書かれている。
