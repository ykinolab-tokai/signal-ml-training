# 第01回 年間方針・役割分担・共通 repo・生成AI運用・コーディング規約

- 対象: B4・M
- 種別: 前期発展枠 / 固定

## この回の目標
- 年間 28 回の流れと、自分の学年で担う役割を説明できる。
- 共通 repo と `submission repo` の使い分けを、提出フローと結びつけて説明できる。
- 生成 AI を使ってよい場面と、申告が必要な理由を説明できる。

## ミニ講義
- この repo の `README.md` は年間方針の source of truth であり、学年ごとの責務、評価、提出先、生成 AI 方針は README を基準に読む。handout ごとのローカルな慣習よりも README を優先する。
- 共通 repo は教材、見本コード、共通資産を置く場所で、通常提出の置き場ではない。通常提出は各自の `submission repo` に残す。両者を混ぜると、教材更新と個人提出の履歴が衝突する。
- 役割分担では、B4 は extension を実装して後輩の視界を広げる側、M は品質確認と支援設計を担う側になる。単に「担当回を書く」だけでなく、自分がどこで価値を出すかまで決めると後続回が進めやすい。

## 演習
### 基礎レベル
1. `README.md` の年間構成、各学年の責務、生成 AI 方針を確認し、`session01_role_plan.md` を作成する。
2. `session01_role_plan.md` には次の 4 見出しをこの順で書く。
   - `## 自分の学年`
   - `## 前期で担当する回`
   - `## 後期で担う支援役割`
   - `## 今年度の重点`
3. `## 自分の学年` には `B4`、`M1`、`M2` のいずれか 1 つを書く。
4. `## 前期で担当する回` には第01回から第14回のうち、自分が主担当または主支援として責任を持つ回番号を書く。
5. `## 後期で担う支援役割` には `mentor`、`reviewer`、`supervisor` のうち該当するものを書く。
6. `## 今年度の重点` には、自分が重視するテーマを 1 つ書き、その理由を 2 行で書く。テーマは `実装品質`、`説明品質`、`レビュー品質`、`再利用可能資産` のいずれかから選ぶ。
7. `session01_ai_ops_checklist.md` を作成し、次の 4 見出しをこの順で書く。
   - `## 共通 repo の役割`
   - `## submission repo の役割`
   - `## 生成AI利用時の申告項目`
   - `## 今年気をつける運用ルール`
8. `## 共通 repo の役割` と `## submission repo の役割` には、それぞれ何を置く場所かを 2 行以内で書く。
9. `## 生成AI利用時の申告項目` には `用途` と `採用箇所` の 2 語を必ず含める。
10. `## 今年気をつける運用ルール` には README から確認したルールを 3 行書く。

### 発展レベル
1. B4 は `session01_role_plan.md` の末尾に `## extension 候補` を追加し、今年度に試したい extension を 1 件書く。あわせて、その extension を扱う前に後輩が理解しているべき前提を 2 項目書く。
2. M は `session01_role_plan.md` の末尾に `## 介入条件` を追加し、review または mentor として介入すべき条件を 3 項目書く。各項目に、なぜその時点で介入するべきかを 1 行添える。
3. `session01_ai_ops_checklist.md` の最後に 2 行追加し、「共通 repo と submission repo を混同した場合に起きる問題」を具体例付きで書く。
4. 生成 AI 利用申告が必要な場面を 2 例、不要な場面を 1 例書き、その違いを 3 行以内で説明する。

## 確認ポイント
- `session01_role_plan.md` に 4 見出しが順番通りあり、担当回と支援役割が具体的に書かれている。
- `session01_ai_ops_checklist.md` に 4 見出しが順番通りあり、`用途` と `採用箇所` が明記されている。
- B4 は extension 候補と前提条件、M は介入条件と理由を書いている。
- repo の使い分けと AI 申告の違いが、単なる用語説明ではなく運用上の理由まで含めて書かれている。
