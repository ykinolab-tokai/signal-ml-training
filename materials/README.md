# materials 概要

このディレクトリには学生配布用 Markdown を置きます。各 handout は、学生がその回の概要をつかみ、演習に着手するための最低限の知識を得て、何をどこまで考えて実装・説明すべきかを一意に解釈できることを目的に書きます。詳細説明は `latex/` や公式 docs に委ね、`materials/` では学習目標、演習に必要な最小知識、課題仕様、確認項目、参照先を明示します。

## 構成

- `spring/`: 前期 14 回の配布資料
- `autumn/`: 後期 14 回の配布資料

## handout の共通方針

- 各回の handout は、`この回の目標`, `ミニ講義`, `演習`, `確認ポイント`, `詰まったときに見る資料` の 5 セクションだけで構成する。
- `この回の目標` には、その回で理解・説明・実装できるようになってほしいことを 2 から 4 項目で書く。
- `ミニ講義` には、演習に手を付けるために最低限必要な知識だけを書く。概念の定義、入力と出力の対応、今回使う式や API の意味、典型的な見落としを短く説明し、詳細な背景説明はここに入れない。
- `演習` は `基礎レベル` と `発展レベル` に分ける。`基礎レベル` は全員が迷わず着手できる必須課題、`発展レベル` は基礎の理解を使って追加の比較・解釈・設計判断を行う課題にする。
- 演習では「小さな例」「簡単な計算」「適宜変更」などの曖昧な表現は使わず、入力、制約、保存物、比較対象、確認すべき観点を明示する。
- ただし、演習を単なる転記課題にしない。配列の値や題材を固定してよいが、学生が結果の解釈、比較、設計上の選択理由、失敗時の切り分けのどれかを自分で考える問いを必ず含める。
- `基礎レベル` では、完了条件を固定しつつも、少なくとも 1 問は「なぜその結果になるか」「どこが入力と出力を決めているか」を説明させる。
- `発展レベル` では、比較条件や評価観点は handout 側で固定し、どの案を採るか、何を優先するか、どの差を重要とみなすかは学生に考えさせる。
- 図・音・中間結果が必要な回では、表示や再生だけで終えず、保存物と確認観点を `演習` または `確認ポイント` に明記する。
- `確認ポイント` には、学生が自己点検できる具体的な観点を書く。shape、値の傾向、保存物の有無、比較結果の説明有無など、手元で確認できる形にする。
- 詳細説明は `latex/markdown/` や公式 docs に委ね、`詰まったときに見る資料` には優先度の高い資料だけを 2 件程度まで明記する。

## 旧資料の吸収方針

- 以前の notebook 群と演習シートにあった具体課題は、現在の回構成に合わせて各 handout に再配置しています。
- 旧 `initial-training/<your_name>/chapterXX` 前提の提出先指定は採用せず、現在の repo と授業運営に合わせて書き換えています。
- B3 の通常提出は学生ごとの private な `submission repo` を前提とし、共通 repo は教材配布と見本コードの参照先として扱います。
- `pull request` は Git 学習回で体験しますが、毎週の通常提出では `edit -> commit -> push` を基本にします。

## 読み方

- B3 向けには `autumn/15` から `autumn/28` を順に使います。
- B4・M 向けには `spring/01` から `spring/14` を使います。
- 前期ローテーション回は、その年度で扱う題材を file 内で明記したうえで配布します。
- handout に書かれていない追加作業をその場で課すのではなく、必要なら handout 自体を書き換えてから配布します。
